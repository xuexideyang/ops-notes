# Redis

## Redis 是单线程还是多线程？
- 6.0 之前：核心命令执行是单线程。
- 6.0 之后：网络 I/O（接收和返回）使用多线程，但命令执行仍是单线程。
- 原因：避免上下文切换和锁竞争，内存操作本身快，瓶颈不在 CPU。

## Redis 使用场景
- **缓存**：将 MySQL 中的数据缓存到 Redis，提高读取速度。
- **分布式锁**：使用 `SETNX` + `EXPIRE` 实现，防止超卖等并发问题。
- **计数器**：如热搜、点赞数，用 `INCR` 命令。
- **排行榜**：使用 Sorted Set（zset），按分数排序。
- **限流**：设置 key 的有效期，在时间内拒绝多余请求。
- **登录态**：将用户 session 存到 Redis，用用户 ID 作为 key 快速查询。

## 缓存穿透、击穿、雪崩
- **穿透**：查询不存在的数据，每次都穿透到数据库。解决：布隆过滤器 / 缓存空对象。
- **击穿**：某个热点 key 过期，大量请求瞬间打到数据库。解决：互斥锁（`SETNX`）/ 逻辑永不过期。
- **雪崩**：大量 key 同时过期，或 Redis 宕机。解决：过期时间加随机值 / 限流降级 / 集群高可用。

## Redis 持久化（RDB vs AOF）
- **RDB**：快照方式，按时间间隔保存。优点：文件紧凑、恢复快、性能高。缺点：两次快照间宕机可能丢数据。
- **AOF**：记录写命令。优点：数据更安全（可每秒同步）。缺点：文件大、恢复慢。
- 生产环境建议 RDB + AOF 同时开启，或开启混合持久化。

## RDB vs AOF 优缺点
- RDB：文件小，恢复快，适合备份，但可能丢数据。
- AOF：数据更安全，文件大，恢复慢。

## 主从复制
- 先建立主从关系，进行数据同步。
- 数据同步根据主从的 offset 判断使用全量还是增量。
- **全量复制**：首次连接或 offset 落后太多时，主库发送 RDB 文件。
- **增量复制**：断线重连后，只同步 offset 之后的新命令。

## 哨兵模式
- 作用：监控、通知、自动故障转移。
- 哨兵之间会选举一个 leader 执行故障转移，将从节点升级为主节点。

## Redis 分布式锁
- 使用 `SETNX` 加锁，设置唯一标识，用 `PX` 设置过期时间。
- 释放时用 Lua 脚本验证唯一标识再删除，防止误删他人的锁。

## Redis 淘汰策略
- `noeviction`：不淘汰，内存满时写入报错。
- `allkeys-lru`：所有 key 中淘汰最近最少使用的。
- `volatile-lru`：设置了过期时间的 key 中淘汰 LRU。
- `allkeys-random`：随机淘汰。
- `volatile-random`：设置了过期时间的 key 中随机淘汰。
- `volatile-ttl`：淘汰存活时间最短的。
- LRU（最近最少使用）vs LFU（最不经常使用）：LRU 看时间，LFU 看访问频率，LRU 可能误伤冷热数据。

## 基本数据类型
- String（字符串）、List（列表）、Hash（哈希）、Set（集合）、Sorted Set（有序集合）

## Redis 集群（Cluster）hash slot
- 共 16384 个槽，每个 key 通过 `CRC16(key) % 16384` 映射到某个槽，每个节点负责一部分槽。

## Redis 事务
- 使用 `MULTI` 开启事务队列，`EXEC` 执行。
- 特点：按顺序执行、不插队、不支持回滚、不隔离。
- `WATCH` 可实现乐观锁。

## Pipeline
- 批量传输多个命令，减少网络往返时间（RTT），但不保证原子性。

## 内存碎片
- 原因：key 的频繁增删改、过期删除导致内存分配不连续。
- 解决办法：重启 Redis（4.0+ 自带内存碎片整理），或调整内存分配器（jemalloc）。
